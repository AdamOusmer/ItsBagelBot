# Native-TLS live NATS acceptance — 2026-07-12

## Topology verified

- Hub and leaf pods have no Linkerd sidecar and carry
  `linkerd.io/inject: disabled` at runtime.
- The benchmark pod was pinned to `node3`; `connz` confirmed the
  `nats-leaf` Service selected `node3--nats-leaf-j8xwf` on the same node.
- Both tested client paths negotiated verified TLS 1.3 with
  `TLS_AES_128_GCM_SHA256` against the fleet CA.
- The BUS leaf link was connected to `nats-2` with approximately 8 ms RTT.

## Production-shaped PubAck result

Parameters: 200,000 messages per endpoint, two independent publisher
connections, 16,384 outstanding PubAcks per connection, unique dedup IDs,
256-byte payload, memory-backed R1 stream, 500 synchronous latency samples.

| Path | Acked | Errors | Throughput | PubAck p50 | p95 | p99 | Max | Gate |
|---|---:|---:|---:|---:|---:|---:|---:|---|
| Direct hub | 200,000 | 0 | 44,327/s | 6.59 ms | 9.20 ms | 15.78 ms | 18.00 ms | pass |
| Node-local leaf → hub | 200,000 | 0 | 24,888/s | 62.92 ms | 92.68 ms | 112.37 ms | 115.50 ms | **fail** |

A quiet 1,000-message follow-up reproduced the latency split: hub p95 was
10.03 ms and leaf p95 was 82.11 ms. The leaf result is therefore not merely a
backlog left by the large run.

## Control result

A Core NATS pub/sub control stayed entirely on the node-local leaf and delivered
all 200,000 messages. One publisher produced 48,904/s and the subscriber drained
43,899/s. This indicates that the leaf client socket and raised pod limits are
working; the failing latency is specific to the JetStream-over-leaf PubAck path.

A separate cross-account RPC test pinned an `ADMIN_RPC` requester to the node3
leaf and a `USERS_RPC` responder to the node2 leaf. It completed 100,000
request/replies at 2,797/s over TLS 1.3. Direct route counters increased by
roughly 200,000 messages (one request plus one reply), proving the standalone
leaf cluster serves cross-node RPC without the hub.

That test also found the old per-account hub remotes forwarding approximately
100,000 redundant request copies toward the hub. The accompanying fix removes
every `*_RPC` hub remote and authorization, leaving only the BUS bridge. The
repeatable `rpc-leaf-direct.sh` test now fails if any RPC hub counter moves.

The large-window CLI stress test also found that 16 clients with 5,000
outstanding publishes each stalled through the leaf, while the direct hub run
completed. Reducing that CLI window to 500 completed at 17,954/s. The reusable
acceptance command uses the actual ingress shape (two × 16,384) and completed
without losing an acknowledgement.

## Broker health during the run

- No slow-consumer or quorum-loss messages appeared in hub or leaf logs.
- The node3 leaf peaked around 817m CPU and 164 MiB, below its 2 CPU / 512 MiB
  limits.
- All temporary streams and benchmark pods were removed after the run.

## Additional migration finding

Leaf logs contained continuous TLS handshake EOF/reset errors from application
pod IPs. Their cadence and the client implementations identified plaintext
failback probes against the now-TLS-only client port. The accompanying fix moves
Go, Elixir, and console probes to the strict-local leaf's HTTP monitor endpoint.

## Decision

Native TLS correctness passes, direct-hub PubAck latency passes, and no messages
were lost. JetStream remains direct-to-hub. The leaf cluster is the independent
Core NATS RPC tier; its direct cross-node path passed, and removing RPC hub
remotes eliminates both the hub dependency and duplicate forwarding.
