# Sesame production acceptance — 2026-07-12

Environment: production NATS 2.14.3 hub over native TLS, temporary R1 memory
streams, four durable input consumers, four publisher connections, two CPU
Sesame shape. Leaves were not used; they remain RPC-only.

## Final replay-free path

Valkey was removed only from transport replay deduplication. Sesame outputs use
a deterministic `Nats-Msg-Id`, confirmed publisher cohorts, and confirmed input
acks. Valkey remains authoritative for cooldowns, loyalty, timers, live state,
and automod reputation.

| node3, 50k commands, 1k channels | 10,920/s | 42.7 ms | 50k unique inputs and outputs, zero errors |
| engine + input, memory output | 20,074/s | 67 us | 50k processed, zero errors |

| node3, simultaneous three-node run | 9,046/s | 47.9 ms | passed |
| worker1, simultaneous three-node run | 13,690/s | 35.9 ms | passed |

The simultaneous fleet drained 60,000 commands in 2.612 seconds: a conservative
aggregate of **22,971 commands/s**. This worst-case test makes every ingress
event a command that emits an output; normal traffic has a much lower command
ratio.

Four consumers x 100 routines is the selected production setting. Raising it to
four x 200 produced 11,149/s on one pod (about 2% more) but raised p95 to 110 ms,
so it was rejected.

## Conclusions

- The engine is not the bottleneck; its measured p95 is 67 us without output
  confirmation.
- Removing the global Valkey replay claim removes a cross-node write from every
  event and avoids worker1's measured ~10.3 ms Valkey-primary RTT.
- Confirmed per-message NATS publication remains the shared ceiling. The current
  official `nats.go` API is used; no Fast-Ingest or atomic wire protocol is
  implemented manually.
- The 22,971/s simultaneous worst-case ceiling is below the 44,327/s ingress
  firehose if every event emits a command. More Sesame pods cannot remove this
  shared NATS confirmation limit.
- KEDA uses 6,000 standard / 1,500 premium lag per replica, based on the slowest
  simultaneous node while retaining the five-pod floor and avoiding breadcrumb
  scale-ups.

## Valkey topology

The external `witness1` Sentinel and its exporter were disabled. Its Tailscale
subnet-router role and `10.0.0.0/16` database route remain active. The four
in-cluster Sentinels report four usable voters, `num-other-sentinels=3`, one
primary, and three online replicas.
